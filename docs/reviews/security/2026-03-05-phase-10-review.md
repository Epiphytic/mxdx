## Adversarial Findings

### Finding 1: CORS allows origin "null" instead of same-origin

- **Severity**: High
- **Location**: `crates/mxdx-web/src/routes/mod.rs:17-19`
- **Description**: The CORS layer is configured with `AllowOrigin::exact("null")`. The `null` origin is sent by pages loaded from `file://` URLs, sandboxed iframes, and certain redirect chains. This does not restrict to same-origin — it restricts to a special origin value that is attacker-controllable. Any sandboxed iframe on any domain can set `Origin: null`. A proper same-origin configuration would either omit CORS entirely (browsers enforce same-origin by default) or use a dynamic origin check that reflects the request origin only when it matches the server's own origin.
- **Recommendation**: Remove the CORS layer entirely. Without a CORS layer, the browser's default same-origin policy blocks all cross-origin requests. Alternatively, if cross-origin access is needed for a specific use case, use `AllowOrigin::exact("http://127.0.0.1:3000")` or a dynamic origin validator.

### Finding 2: SRI verification is a no-op — server never sends X-Content-Hash

- **Severity**: Medium
- **Location**: `crates/mxdx-web/static/sw.js:66-68`, `crates/mxdx-web/src/routes/static_files.rs`
- **Description**: The service worker's `verifyIntegrity` function checks for an `X-Content-Hash` response header. If the header is absent, the function returns the response unverified (line 68: `return response`). The static file handlers in `static_files.rs` never set `X-Content-Hash`. As a result, SRI verification is dead code — every cached response passes without integrity checking. This means a cache-poisoning attack (e.g., via a compromised network on first load, or a bug in the service worker cache) would persist undetected.
- **Recommendation**: Either (a) add `X-Content-Hash: sha256-<hex>` headers to static file responses computed at build time, or (b) embed a hash manifest directly in `sw.js` (generated by xtask) and verify against it. Option (b) is more robust since response headers can be stripped by proxies.

### Finding 3: HTMX partials lack same-origin enforcement on server side

- **Severity**: Medium
- **Location**: `crates/mxdx-web/src/routes/dashboard.rs:12`
- **Description**: The `/dashboard` endpoint returns HTMX partial HTML to any request. While the current dashboard is read-only (no mutations), a cross-origin page could embed the dashboard response via `<img>`, `<script>`, or fetch (if CORS permits). The service worker checks `HX-Request` header (sw.js:33) to decide caching strategy, but the server does not. An attacker could craft cross-origin requests that fetch dashboard partials to extract launcher IDs, hostnames, CPU/memory stats, and other fleet metadata.
- **Recommendation**: Add server-side middleware that checks for the `HX-Request` header on partial endpoints (or a `Sec-Fetch-Site: same-origin` check). Return 403 for cross-origin requests to HTMX partials.

### Finding 4: SSE endpoint has no authentication or rate limiting

- **Severity**: Medium
- **Location**: `crates/mxdx-web/src/routes/sse.rs:14`
- **Description**: The `/sse/launchers` endpoint streams real-time launcher updates to any client that connects. There is no authentication, no session check, and no rate limiting. An attacker who discovers the web server address can subscribe to the SSE stream and receive continuous fleet telemetry (launcher IDs, hostnames, CPU usage, memory stats). Additionally, each SSE connection holds a broadcast channel receiver. Without connection limits, an attacker could open many SSE connections to exhaust server memory via broadcast channel buffering.
- **Recommendation**: (a) Bind the server only to localhost (currently done in `main.rs:14` — good). (b) Add a connection limit middleware (e.g., `tower::limit::ConcurrencyLimitLayer`). (c) When authentication is added in a future phase, gate SSE access behind it.

### Finding 5: Service worker caches cross-origin fetch failures as undefined

- **Severity**: Low
- **Location**: `crates/mxdx-web/static/sw.js:43`
- **Description**: In the network-first path (HTMX requests), when `fetch()` fails, the service worker falls back to `caches.match(event.request)`. If no cache entry exists, this returns `undefined`, which will cause the browser to receive no response (network error). This is correct fail-closed behavior. However, the cache-first path (line 54) does not have an equivalent `.catch()` handler — if both the cache miss and the network fetch fail, the promise rejects unhandled.
- **Recommendation**: Add a `.catch()` to the cache-first fetch chain (line 54-57) that returns a fallback offline page or a proper error response.

### Finding 6: Hardcoded bind address in main.rs

- **Severity**: Low
- **Location**: `crates/mxdx-web/src/main.rs:14`
- **Description**: The server binds to `127.0.0.1:3000`. The localhost binding is good for security (not network-accessible). However, port 3000 is hardcoded. If the port is in use, the server panics. This is a reliability issue, not directly a security issue, but in production the bind address should be configurable.
- **Recommendation**: Accept bind address from environment variable or config file. This also enables the OS-assigned port pattern (`127.0.0.1:0`) used in tests.

### Finding 7: style-src 'unsafe-inline' in CSP

- **Severity**: Low
- **Location**: `crates/mxdx-web/src/routes/mod.rs:24`
- **Description**: The CSP includes `style-src 'self' 'unsafe-inline'`. While this is common for HTMX applications (inline style attributes for dynamic updates), it does weaken CSP protection against style-based data exfiltration attacks (CSS injection). The `script-src 'self'` directive properly blocks inline scripts, which is the more critical protection.
- **Recommendation**: If inline styles are not actually needed, remove `'unsafe-inline'` from `style-src`. If they are needed for HTMX class swaps, document this as an accepted trade-off.

## Summary

Phase 10 security posture is **fair**. The CSP header correctly prevents inline scripts, the app is stateless with no Matrix credentials, and the server binds to localhost only. However, three issues need attention:

1. **CORS origin "null" is insecure** (High) — should be removed entirely so browser default same-origin policy applies.
2. **SRI verification is dead code** (Medium) — the server never sets the hash header that the service worker checks.
3. **HTMX partials lack server-side origin checking** (Medium) — fleet metadata is accessible to any caller.

The SSE endpoint's lack of connection limits is a moderate DoS risk, mitigated by the localhost-only binding. Once CORS is fixed and SRI is wired up, the web layer will meet the Phase 10 security requirements.
